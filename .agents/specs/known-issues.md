## 1. 냉장고 파먹기 — 재료 동의어/유의어 미처리

**증상**: "계란"과 "달걀", "양파"와 "어니언", "소고기"와 "쇠고기" 등 같은 재료가 다른 문자열로 저장되어 있으면 매칭 실패.

**원인**: `search_recipes_by_ingredients` RPC가 입력 문자열과 레시피 재료 문자열을 단순 `ILIKE` 비교.

**해결 방향 (미결)**:
- 옵션 A: 동의어 사전 테이블(`ingredient_synonyms`) 생성 후 RPC에서 조인
- 옵션 B: 재료 정규화 함수를 통해 저장/검색 시 표준화 (예: 모두 "달걀"로 통일)
- 옵션 C: AI 임베딩 기반 유사도 검색 (비용 높음, 장기 과제)

**담당**: Backend

## 2. 냉장고 파먹기 — 로그인 유저의 비공개 레시피 미노출

**증상**: 로그인한 유저가 직접 작성한 비공개 텍스트 레시피는 냉장고 파먹기 결과에서 나오지 않음.

**원인**: `POST /recipes/fridge-search` 에서 `p_public_only: True` 하드코딩 (비로그인 대응용).

**위치**: `backend/main.py` 라인 1217

**해결 방향**:
- 로그인 유저면 `p_public_only: False` + `p_user_id` 파라미터 전달 → RPC에서 `is_public = true OR author_user_id = p_user_id` 조건 추가
- `get_current_user_optional` 의존성 추가 필요

**담당**: Backend

## 3. 칼로리 정보 불일치 — AI 추출 vs AI 어시스턴트

**증상**: 레시피 카드에 표시되는 칼로리(AI 추출 시 계산)와 어시스턴트가 답변하는 칼로리가 다를 수 있음.

**원인**: `POST /ai/chat` 의 시스템 프롬프트에 저장된 `calories` 값이 전달되지 않아 AI가 독립적으로 재계산.

**추가 맥락**: 칼로리가 1인분 기준인지 전체 기준인지 레이블도 불명확.

**해결 방향**:
- `/ai/chat` 시스템 프롬프트에 `calories` 값과 단위(예: "1인분 기준 약 {n}kcal") 포함
- 레시피 추출 프롬프트에서 "1인분 기준 칼로리"임을 명시하도록 수정
- 위 두 작업 모두 `backend/ai_engine.py` 수정 필요

**담당**: Backend

## 4. 칼로리 추출값 과소평가 — AI 추정 방식의 한계

**증상**: AI가 추출한 1인분 칼로리가 실제보다 현저히 낮게 나옴.
예) 목살고추장찌개 → 400kcal, 육회 → 260kcal (실제보다 낮은 수준).

**원인**:
1. AI가 재료별 칼로리를 실제 합산하지 않고 "대략 추정" 방식으로 응답
2. "1인분" 기준이 모호 — 한국 성인 실식사량이 아닌 서양 영양 데이터 기준으로 계산할 가능성
3. 기름·소스·양념류 칼로리(고추장 1T ≈ 30kcal, 참기름 1T ≈ 120kcal) 과소반영
4. servings로 나누기 전 전체 칼로리 자체를 낮게 산정

**해결 방향**:
- 프롬프트에 계산 방법 구체화: "각 재료 사용량 기준으로 합산 후 servings로 나눌 것"
- 한국 성인 실식사량 기준 명시 (찌개 1인분 ≈ 300~400g, 볶음 1인분 ≈ 200~300g)
- 기름·소스류 칼로리 반드시 포함 명시
- 프론트엔드에 "1인분 약" 라벨 추가

**위치**: `backend/ai_engine.py` `_build_prompt()` 및 `_build_text_prompt()` 8번 항목

**담당**: Backend
