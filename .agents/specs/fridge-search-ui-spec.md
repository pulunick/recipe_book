## 1. UI 배치 방식 결정

### 1.1 후보 비교

| 방식 | 장점 | 단점 | 판정 |
|------|------|------|------|
| **A. `/fridge` 별도 라우트** | 화면 전체 활용, 복잡한 UI 수용 가능, 탐색 탭과 독립적 | 진입 경로 추가 필요, 바텀 네비 탭 외의 페이지로 존재감 약함 | **채택** |
| B. 탐색 탭 내 토글/스위치 | 한 화면에서 전환 가능 | 탐색 탭이 이미 복잡 (검색+필터+카테고리+랜덤배너+그리드), 상태 관리 복잡 | 기각 |
| C. 바텀 네비 탭 추가 (6탭) | 상시 접근 가능 | 5탭이 UX 한계, 중앙 [+] 버튼 밀려남, 라벨 겹침 | 기각 |

### 1.2 결정: `/fridge` 별도 페이지 + 탐색 탭 배너 진입

**근거:**
1. 탐색 탭(`/`)은 이미 검색바 + 소스 필터 + 카테고리 칩 + 필터 뱃지 + 랜덤 배너 + 카드 그리드로 밀도가 높음. 여기에 재료 입력 UI까지 넣으면 과부하.
2. 냉장고 파먹기는 "재료 입력 → 결과 탐색"이라는 독립적 플로우를 갖고 있어 별도 화면이 자연스러움.
3. 탐색 탭의 "오늘 뭐먹지" 배너 아래에 "냉장고 파먹기" 배너를 배치하면 발견성(discoverability) 확보.
4. 향후 Flutter 앱에서도 동일 라우트로 매핑 가능.

## 2. 진입점 및 플로우

### 2.1 진입점

**탐색 탭(`/`) 배너:**
- "오늘 뭐먹지" 배너 바로 아래에 "냉장고 파먹기" 배너 배치
- 검색 중이거나 필터 활성화 시 랜덤 배너와 함께 숨김
- 배너 클릭 시 `/fridge`로 이동

### 2.2 사용자 플로우

```
탐색 탭(/) → "냉장고 파먹기" 배너 탭
    → /fridge 페이지 이동
    → 재료 입력 (칩 태그, 추천 칩)
    → "검색" 버튼 탭
    → 결과 카드 목록 (매칭 점수 포함)
    → 카드 탭 → /recipe/[id] 상세 보기
    → 보관함 추가 가능
```

## 4. 상세 동작 명세

### 4.1 재료 입력

- **입력 방식**: 텍스트 입력 → Enter 키 또는 쉼표(,)로 칩 추가
- **최대 개수**: 15개 (초과 시 안내 메시지)
- **최소 글자**: 2자 이상 (1자는 무시, RPC와 동일)
- **중복 방지**: 동일 재료명 입력 시 무시
- **삭제**: 칩 내 X 버튼 또는 Backspace(빈 입력 상태에서)
- **추천 칩**: 탭하면 즉시 추가, 이미 추가된 재료는 비활성화(회색) 표시

### 4.2 추천 재료 목록

```
["김치", "돼지고기", "두부", "계란", "양파", "파", "마늘", "간장",
 "고추장", "감자", "당근", "버섯", "치즈", "소시지", "참치캔", "라면"]
```

### 4.3 검색 버튼

- 재료 0개: 버튼 비활성화
- 재료 1개 이상: "레시피 찾기 (N)" 형태로 재료 수 표시
- 탭 시 `POST /recipes/fridge-search` 호출

### 4.4 결과 목록

- 카드 형태: 가로 풀 너비 리스트 (그리드 아닌 세로 리스트)
- 매칭 점수 프로그레스 바: `match_score`% 표시, terracotta 색상
- 일치 재료: `matched_ingredients` 배열을 칩으로 표시
- 정렬: 매칭 점수 내림차순 (API 기본값)
- 보관함 추가: 탐색 탭과 동일 로직 (`saveToCollection`)
- 카드 탭: `/recipe/{video_id ?? id}` 상세 페이지 이동
- 결과 0건: "일치하는 레시피가 없어요. 재료를 줄이거나 다른 재료를 입력해보세요." 안내

### 4.5 비로그인 처리

- `/fridge` 페이지 자체는 **비로그인 접근 허용** (공개 레시피 검색이므로)
- 보관함 추가 시에만 로그인 필요 (LoginModal 표시)

## 6. 백엔드 연동

### 6.1 기존 명세 확인 (feature-roadmap-v07.md 기능 4)

**RPC 함수**: `search_recipes_by_ingredients(p_ingredients, p_limit, p_public_only)` -- 이미 확정

**API 엔드포인트**: `POST /recipes/fridge-search` -- 이미 확정

### 6.2 추가 필요 사항

**없음.** 백엔드 API 계약은 기존 명세 그대로 사용:
- 요청: `{ "ingredients": ["김치", "돼지고기"], "limit": 10 }`
- 응답: `FridgeSearchResultItem[]` (id, title, category, cooking_time, difficulty, servings, video_id, channel_name, source, collection_count, match_score, matched_ingredients)

단, 프론트에서 `my_collection_id` 정보가 필요한 경우(보관함 추가 버튼 상태 표시), 백엔드 응답에 `my_collection_id` 필드 추가를 검토할 수 있음. 이는 `checkCollection` API를 별도 호출하거나, RPC에 user_id JOIN을 추가하는 방식으로 해결 가능. **우선 구현 시 `checkCollection` 별도 호출 방식 권장** (RPC 수정 최소화).

## 7. 디자인 가이드

### 7.1 배너 스타일

- 기존 "오늘 뭐먹지" 배너와 **동일한 높이/padding/border-radius**
- 배경: 연한 파랑 그라데이션 (`color-mix(in srgb, #4A90D9 12%, white)` 계열) -- 냉장고 차가운 느낌
- 아이콘: 냉장고 이모지 또는 눈결정 이모지
- 우측: 화살표(`>`) 버튼

### 7.2 `/fridge` 페이지 스타일

- 기존 앱 디자인 시스템 준수: `--color-terracotta`, `--color-warm-brown`, `--color-cream` 등
- 칩 스타일: 기존 `.chip` 클래스와 동일
- 프로그레스 바: `--color-terracotta` 배경, 높이 6px, border-radius 3px
- 카드: 기존 `.recipe-card`와 유사하되 가로 풀 너비 리스트 형태

### 7.3 먹당 캐릭터 활용

- 안내 영역에 `/static/meokdang.png` 이미지 활용 가능 (존재 시)
- 빈 상태/결과 없음에서도 캐릭터 이미지 활용
